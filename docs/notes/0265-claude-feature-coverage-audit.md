## Five high-level findings

1. Yoke's Claude Python SDK coverage is strong for runnable v1 primitives: one-shot/session runs, hooks, permission callbacks, skills, declared subagents, portable workflows, fork, and stored session list/read/rename/tag.
2. Yoke correctly separates provider-native Claude features from translated ones: Claude subagents/hooks/skills/permissions are native SDK fields; portable workflows and run goals are Yoke-level translations.
3. Native Claude TypeScript Workflow execution remains the largest feature gap. Yoke can model/bundle Claude workflow scripts, but the runnable Claude adapter rejects native workflow execution.
4. Claude subagent history is underexposed. The SDK can list subagent ids and read subagent messages, including store-backed variants, but Yoke's session history API only reads main session messages.
5. MCP is mostly passthrough today. Claude has rich stdio/SSE/HTTP/SDK MCP config plus status APIs; Yoke lacks a provider-neutral MCP config/status surface and folder syntax.
