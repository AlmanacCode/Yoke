# 0266 - Codex feature coverage audit

Date: 2026-07-04

Scope: local `/Users/rohan/Desktop/Projects/openai-codex`, its `sdk/python`, `sdk/typescript`, `codex-rs/app-server`, local `claude-agent-sdk-python` where useful for equivalents, and `/Users/rohan/Desktop/Projects/Yoke`. Focused only on goals, subagents/collab agents, skills/instructions, MCP/tool config, hooks, sessions, event streaming, approval/sandbox/options, and Codex Python SDK vs app-server differences.

## Most important findings

1. **Codex app-server is the only Codex surface with real native goals.** App-server documents `thread/goal/set`, `thread/goal/get`, `thread/goal/clear`, and goal update/clear notifications (`openai-codex/codex-rs/app-server/README.md:153-157`). Yoke correctly treats `codex_app_server` goal, goal loop, mutable goal, and readable goal as native (`Yoke/src/yoke/surfaces.py:1496-1503`) and has adapter entrypoints for `goal_loop`, `get_goal`, `set_goal`, and `clear_goal` (`Yoke/src/yoke/providers/codex_app_server.py:202,370-379`). The Codex Python SDK does not expose native goal methods in its public convenience API; Yoke marks SDK goals as compiled/unsupported for native read/write (`Yoke/src/yoke/surfaces.py:1339-1350`).

2. **Yoke’s app-server coverage is deeper than its Codex Python SDK coverage, and that difference is real upstream.** The Python SDK public API exposes thread start/list/resume/fork/archive/unarchive/models, run/turn/read/name/compact, sandbox, stream/steer/interrupt, and `SkillInput` (`openai-codex/sdk/python/docs/api-reference.md:68-74,153-162,187-232,240-243`). App-server exposes more product-control surface: experimental permissions profiles, runtime workspace roots, selected capability roots, collaboration mode, thread settings, goal APIs, session listing filters for spawned children, and background terminal controls (`openai-codex/codex-rs/app-server/README.md:140-171`). Yoke reflects this: `codex_app_server` is native for sessions, streaming, Codex permissions, collab agent tools, collaboration mode, skills, plugins, MCP, goals, interrupt, fork, and experimental API (`Yoke/src/yoke/surfaces.py:1410-1521`), while `codex_python_sdk` leaves many of those unsupported, compiled, or unknown (`Yoke/src/yoke/surfaces.py:1291-1370`).

3. **Subagents are split into two different concepts.** App-server has native collab-agent tool events such as `spawnAgent` and spawned thread lineage (`openai-codex/codex-rs/app-server/README.md:144`; `openai-codex/sdk/python/src/openai_codex/generated/v2_all.py:6624-6668`). Yoke maps app-server collab tool calls to agent tool events and marks `COLLAB_AGENT_TOOLS` native (`Yoke/tests/test_codex_app_events.py:15-60,63-106`; `Yoke/src/yoke/surfaces.py:1476-1479`). But Yoke-declared subagents are still compiled into instructions, not native app-server collab agents (`Yoke/src/yoke/surfaces.py:1467-1475`). That distinction should stay explicit.

4. **Skills and MCP are partially typed, not fully managed.** App-server supports selected capability roots, plugin/standalone skill roots, MCP from selected plugins, and OpenAI MCP form elicitation capability negotiation (`openai-codex/codex-rs/app-server/README.md:89-94,140`). Yoke wires packaged skills through native extra roots (`Yoke/src/yoke/providers/codex_app/skills.py:11-49`; `Yoke/src/yoke/surfaces.py:1484-1495`) and normalizes MCP tool calls / elicitation requests as events (`Yoke/src/yoke/providers/codex_app/events.py:464-535,587-590,707-710`). The Python SDK adapter explicitly says SDK `SkillInput` is not wired yet and typed MCP setup is not exposed (`Yoke/src/yoke/surfaces.py:1323-1338`).

5. **Hooks are the shakiest Codex app-server coverage claim.** Upstream generated protocol includes hook notifications and hook list types (`openai-codex/sdk/python/src/openai_codex/generated/notification_registry.py:87-88`; `openai-codex/sdk/python/src/openai_codex/generated/v2_all.py:4909-4914,5877-5945,7347,7584-7588`), and app-server mentions `SessionStart` hook source control via `sessionStartSource` (`openai-codex/codex-rs/app-server/README.md:140`). Yoke has rich typed Claude hooks (`Yoke/src/yoke/options.py:281-377`) and marks Codex app-server hooks as native (`Yoke/src/yoke/surfaces.py:1494`), but I did not find an equivalent typed Codex hook configuration surface in Yoke; currently hook notifications would be at best generic stream events unless adapter code is added.

## Coverage summary

- Strong: app-server sessions, streaming, session list/read/resume/compact/rename, goals, goal loop, fork, interrupt, request events, approval/sandbox/permissions lowering, collab-agent event normalization.
- Partial: skills, plugins, MCP. Yoke can wire skill roots and normalize MCP activity, but it does not manage full plugin install/import or provide typed MCP server configuration for the SDK surface.
- Weak/gap: Codex hooks as a typed Yoke concept, native mapping from Yoke-declared subagents into app-server collab agents, and Python SDK access to app-server-only controls such as native goals and experimental app-server fields.
