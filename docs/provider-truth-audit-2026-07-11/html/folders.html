Folder placement

Provider folders are resolved relative to the provider's working project

For a project-scoped definition, .codex or .claude belongs at the root of the repository passed as the agent's working directory. CodeAlmanac complicates this because its definitions belong to the product, not to every user's repository.

01 — Canonical source

What CodeAlmanac owns

codealmanac/ └─ src/codealmanac/agents/ # canonical Yoke collection ├─ yoke.yaml ├─ build/ ├─ ingest/ └─ garden/
Source of truth · agent intentThe packaged Yoke collection owns the definitions. Generated provider files are projections and must never become a second hand-edited source.

02 — Codex

Where `.codex` goes

If Codex starts with cwd=/work/acme-api, project custom agents must be discoverable here:

/work/acme-api/ # Codex thread cwd / project root ├─ .codex/ │ ├─ config.toml │ └─ agents/ │ ├─ build-writer.toml │ └─ reviewer.toml └─ .agents/skills/ # shared skill convention
Why — per the provider surfaceCodex app-server starts a thread with an explicit cwd. Native collaboration events prove child execution, but do not prove a Yoke declaration was registered. Source: OpenAI · app-server

03 — Claude

Where `.claude` goes

Claude has two legitimate paths. Programmatic subagents require no file. Project filesystem agents and skills live under the Claude working project:

/work/acme-api/ # Claude SDK cwd / project root └─ .claude/ ├─ agents/ │ └─ reviewer.md └─ skills/ └─ source-grounding/ └─ SKILL.md

Yoke's active Claude adapter passes subagents through the SDK agents parameter and passes the packaged Yoke root as a local plugin path for skills. It does not need to copy `.claude/agents` into the user's repository.

Why — per the docsClaude recommends programmatic subagents for SDK applications. Skills are filesystem artifacts discovered from project/user locations or loaded from a local plugin path. Claude · SDK subagents · SDK skills

04 — CodeAlmanac

The placement conflict

site-packages/codealmanac/agents/ # product-owned definitions /Users/rohan/Projects/customer-repo/ # harness cwd

For Claude SDK this is fine: definitions travel through SDK objects and plugin paths. For true Codex custom agents, generated files must be visible to the Codex project rooted at customer-repo. Writing them under the CodeAlmanac source checkout does nothing when Codex is working elsewhere.

05 — Recommendation

Do not silently write user repositories

Current gapYoke has no implemented per-run custom-agent materialization layer for Codex app-server. CodeAlmanac therefore does not currently get true named Codex custom agents from packaged Yoke subagent folders.

Codex's configuration model accepts an absolute agents.<name>.config_file, so Yoke can register generated role files without creating customer-repo/.codex. However, registration is only half the contract: current app-server live tests could not prove that the model can select that named role. Yoke should keep prompt compilation as an explicitly weaker fallback until named selection and child-model application are observed end to end.
